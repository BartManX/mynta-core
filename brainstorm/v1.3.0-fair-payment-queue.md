# v1.3.0 Consensus Upgrade: Fair Masternode Payment Queue

## The Problem

`CalcScore()` in `evo/deterministicmns.cpp` feeds `blocksSincePayment` into a
cryptographic hash alongside `proTxHash` and `blockHash`. Because SHA256 output
is uniformly distributed regardless of input magnitude, the "payment history"
component has **zero effect on priority**. Payment selection is a pure lottery
each block.

```cpp
// CURRENT (broken) -- score is random regardless of wait time
CHashWriter hw(SER_GETHASH, PROTOCOL_VERSION);
hw << proTxHash;
hw << blockHash;
hw << blocksSincePayment;   // changes the hash, does NOT lower it
return UintToArith256(hw.GetHash());
```

### Observed impact (mainnet, height ~51973)

- 217 registered masternodes, expected cycle ~217 blocks
- 47 masternodes have waited 300+ blocks without payment
- Worst case: 854 blocks (4x expected cycle)
- One operator with 36 MNs (~16.6% of network) gets paid roughly every 6 blocks
  by pure statistical advantage, while single-MN operators can wait indefinitely

## The Fix

Replace the single hash-based lottery with a **two-tier deterministic queue**:

1. **Primary sort: `lastPaidHeight` ascending** (FIFO -- longest wait = highest priority)
2. **Tiebreaker: hash(proTxHash, blockHash)** for deterministic ordering among
   MNs with the same `lastPaidHeight`

This is how Dash's payment queue works. Every masternode is guaranteed payment
within approximately one full cycle (N blocks for N masternodes).

```
// PROPOSED -- FIFO queue with hash tiebreaker
//
// Phase 1: Find the lowest lastPaidHeight among all valid MNs
// Phase 2: Among MNs sharing that lastPaidHeight, pick the one
//           with the lowest hash(proTxHash, blockHash)
//
// Result: strict round-robin with deterministic tie resolution
```

## Activation Strategy: BIP9 Version Bit Signaling

This is a **consensus change** -- the payment selection function is called during
both block creation (miner.cpp) and block validation (validation.cpp). All nodes
must agree on who gets paid or blocks get rejected.

### Deployment definition

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Deployment name | `DEPLOYMENT_MN_FAIR_QUEUE` | Descriptive |
| Version bit | 11 | First unused bit after existing deployments |
| Start time | TBD | After binaries are distributed |
| Timeout | TBD | Generous window for miner adoption |
| Confirmation window | 2016 (default) | ~33.6 hours at 1-min blocks |
| Activation threshold | 1613 (default, ~80%) | Matches existing mainnet params |

### BIP9 state machine flow

```
DEFINED ──(startTime reached)──> STARTED ──(80% signal in window)──> LOCKED_IN ──(next window)──> ACTIVE
                                    │
                                    └──(timeout reached)──> FAILED
```

### Files to modify

1. **`consensus/params.h`** -- Add `DEPLOYMENT_MN_FAIR_QUEUE` to `DeploymentPos` enum
2. **`versionbits.cpp`** -- Add deployment info entry
3. **`chainparams.cpp`** -- Configure bit, start time, timeout for mainnet/testnet/regtest
4. **`evo/deterministicmns.cpp`** -- Dual-path `CalcScore()` / `GetMNPayee()`:
   - Before activation: current (broken) hash-only logic
   - After activation: FIFO + hash tiebreaker
5. **`validation.cpp`** -- Pass deployment state into payment validation
6. **`miner.cpp`** -- No change needed (already uses `ComputeBlockVersion()`)

### Code sketch for `GetMNPayee()` post-activation

```cpp
CDeterministicMNCPtr CDeterministicMNList::GetMNPayee(
    const uint256& blockHash, int currentHeight, bool fFairQueue) const
{
    auto validMNs = GetValidMNsForPayment(currentHeight);
    if (validMNs.empty()) return nullptr;

    if (!fFairQueue) {
        // Legacy path (pre-activation) -- pure hash lottery
        // [existing code unchanged]
    }

    // --- Fair queue path (post-activation) ---

    // Find the minimum lastPaidHeight (longest-waiting MNs)
    int minLastPaid = INT_MAX;
    for (const auto& mn : validMNs) {
        int lph = mn->state.nLastPaidHeight > 0
                    ? mn->state.nLastPaidHeight
                    : mn->state.nRegisteredHeight;
        if (lph < minLastPaid) minLastPaid = lph;
    }

    // Collect all MNs at that minimum (the "due" set)
    std::vector<CDeterministicMNCPtr> dueMNs;
    for (const auto& mn : validMNs) {
        int lph = mn->state.nLastPaidHeight > 0
                    ? mn->state.nLastPaidHeight
                    : mn->state.nRegisteredHeight;
        if (lph == minLastPaid) dueMNs.push_back(mn);
    }

    // Deterministic tiebreaker among the due set
    CDeterministicMNCPtr winner = nullptr;
    arith_uint256 lowestHash;
    bool first = true;
    for (const auto& mn : dueMNs) {
        CHashWriter hw(SER_GETHASH, PROTOCOL_VERSION);
        hw << mn->proTxHash;
        hw << blockHash;
        arith_uint256 h = UintToArith256(hw.GetHash());
        if (first || h < lowestHash) {
            winner = mn;
            lowestHash = h;
            first = false;
        }
    }
    return winner;
}
```

## Rollout Plan

1. **Implement and test on regtest** -- Verify fair rotation with N masternodes
2. **Deploy to testnet** -- BIP9 with short window for quick activation
3. **Release v1.3.0 binaries** -- Announce upgrade, set mainnet start time
4. **Miners upgrade and signal** -- 80% threshold over 2016-block window
5. **Activation** -- One window after lock-in, new payment logic enforced

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Chain split if nodes disagree on activation | BIP9 ensures supermajority before enforcement |
| Miners slow to upgrade | Generous timeout window; pool operators notified directly |
| Edge case: all MNs registered at same height, never paid | `nRegisteredHeight` used as fallback -- hash tiebreaker resolves |
| Operator with 36 MNs sees reduced frequency | Expected and correct -- each MN still gets 1 payment per cycle |

## Open Questions

- Should we bundle other consensus fixes in this deployment or keep it isolated?
- Confirmation window override -- 2016 blocks is ~33h, is that appropriate for
  a network this young or should we use a shorter window?
- Should the timeout be generous (months) or tight (weeks)?
